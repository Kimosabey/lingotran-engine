How it works

Only pay a model where it must think

The one rule. A model is used only for eyes (reading a scan) or judgement (finding an omission, classifying). Moving data around is plain Python — free.
Orchestration

One page, end to end

Work moves left to right. Each step writes a file the next step reads, so any step re-runs on its own and the whole job resumes after an interruption.

Example — Kursbuch page 66 (a chat screen). Python renders the image → Opus reads the picture and types the 12 chat bubbles in order → Opus reads it again as a sceptic and confirms nothing dropped → a cheap text model tags it dialogue · communication and pulls out the items → Python writes the sheets.

The roles

Think of it as a small team — a few expensive "thinkers" and a lot of cheap, reliable "machines."

Who does what

Which model, and why

Three tiers. The engine runs end-to-end on one model (Opus 4.8); the cheaper text model is purely a cost saver on the easy work.

Detail

Thirteen layers, one job each

Every layer takes a defined input, does exactly one job, and writes an artifact the next layer reads. Any layer can be re-run alone.

Cost

Where the usage goes

On subscription accounts you don't pay per token — you spend a usage window. So "expensive" means how fast a run eats the window. Reading scans dominates; a chunk is avoidable.

Making it leaner + more robust
Biggest single win: detect a text layer first. Digital-born PDFs already contain their text — pull it out with Python for free and skip vision entirely. Vision is only needed for genuine scans.
